# EmpValue-AI 安全合规白皮书

> **版本：** v1.0  
> **适用对象：** 企业安全合规、法务、HR、系统管理员  
> **目标：** 说明 EmpValue-AI 在数据安全、内容安全、法律合规、公平性审计等方面的设计原则与落地措施

---

## 一、合规立场声明

EmpValue-AI 是一款**决策支持工具**，而非人事决策主体。系统所有输出均标注“AI 生成，仅供参考，最终决策由人作出”。任何涉及晋升、调岗、绩效改进、奖惩等人事决策，必须由具备权限的主管或 HR 人工确认后方可生效。

---

## 二、数据安全

### 2.1 数据分类

| 数据类型 | 敏感等级 | 示例 |
|---|---|---|
| 原始输入 | 高敏感 | 日报、任务进度、代码提交、会议记录、截图、语音 |
| 评估结果 | 高敏感 | 员工视图、管理视图、风险标记、隐藏问题 |
| 审计日志 | 高敏感 | 操作人、操作时间、IP、审批记录、申诉记录 |
| 配置信息 | 中敏感 | API Key、数据库连接串、模型档位 |

### 2.2 传输与存储加密

- **传输加密：** 全站强制 TLS 1.3，禁止明文 HTTP。
- **存储加密：**
  - 数据库静态数据使用 AES-256 加密（依赖 PostgreSQL/SQLite 透明加密或文件系统加密）；
  - 向量库与对象存储中的原始附件使用服务器端加密；
  - 密码使用 bcrypt 哈希存储，不可逆向。

### 2.3 租户隔离

- 数据库按企业/租户独立 Schema 或独立实例；
- 向量库集合（collection）按租户隔离；
- API 认证基于 JWT，所有数据访问均经过 RBAC 校验。

### 2.4 本地部署选项

企业可选择**本地模式**：模型运行在 LM Studio / Ollama 上，原始数据不出内网，满足高保密行业要求。

## 三、内容安全与护栏

### 3.1 输入护栏

系统在后端对输入进行多层校验：

| 风险 | 防护措施 |
|---|---|
| Prompt 注入 | 输入长度限制、关键词过滤、结构化校验 |
| 恶意文件 | 文件类型白名单、大小限制、病毒扫描（建议接入企业杀毒） |
| 超大输入 | 单条内容长度与附件总大小限制 |
| 非工作相关内容 | 可选启用内容相关性过滤 |

### 3.2 输出护栏

| 风险 | 防护措施 |
|---|---|
| PII 泄露 | 输出脱敏模块，对手机号、身份证号、银行卡等敏感实体进行 redaction |
| 歧视性表述 | 禁用基于性别、年龄、地域、民族等的负面归因 |
| 极端负面措辞 | 员工视图禁用“差、懒、无能、失败”等词汇 |
| 直接人事处置 | 禁止输出“建议辞退”“建议降薪”等结论 |

### 3.3 幻觉控制

- 强制每个维度得分必须引用原始输入中的具体证据；
- `confidence_score` 低于阈值时触发人工复核；
- 所有输出保留模型名、Prompt 版本、原始数据引用，便于追溯。

## 四、权限与访问控制

### 4.1 角色体系

| 角色 | 可访问数据 |
|---|---|
| employee | 仅自己的员工视图 |
| manager | 团队成员管理视图、审批 |
| hr | 全部评估复核、审计日志 |
| admin | 系统配置、全部审计日志 |

### 4.2 关键访问规则

- 员工视图与管理视图在同一推理中生成，但分开展示；
- 管理视图仅限 manager/hr/admin 查看，并记录审计日志；
- 员工只能查看自己的评估，不能查看他人；
- 关键页面建议增加水印，防止截图泄露。

### 4.3 审计日志

系统记录以下操作：

- 登录/登出/登录失败
- 原始输入提交
- 评估创建、审批、驳回、申诉、重新评估
- 管理视图查看
- 模型档位切换
- 管理后台审计日志查询

审计日志不可篡改，保留期限建议 ≥ 5 年。

## 五、人机协同与人工最终决策

### 5.1 审批状态机

```
RAW_DATA → AI_PROCESSING → AI_DRAFTED → MANAGER_REVIEW → HR_AUDIT → APPROVED
                                              ↓
                                        FEEDBACK_COLLECT → 申诉 → 重新评估
```

- **AI_DRAFTED：** AI 生成草稿，不可直接发布；
- **MANAGER_REVIEW：** 主管必须审阅并决定通过/驳回/提交 HR 复核；
- **HR_AUDIT：** 异常评估自动进入 HR 复核；
- **APPROVED：** 人工确认后，员工视图方可对员工可见。

### 5.2 申诉机制

- 员工可对 approved / rejected 的评估提出申诉；
- 申诉触发评估回到 manager_review 状态；
- 申诉记录进入审计日志。

## 六、公平性与偏见控制

### 6.1 设计原则

- 评估基于行为和产出，而非人口统计学特征；
- Prompt 中禁止要求模型使用性别、年龄、地域等信息进行评价；
- 所有结论必须有可追溯的证据。

### 6.2 公平性审计

建议企业每季度执行：

- 按性别、年龄段、职级抽样检查评分分布；
- 对比同一岗位不同群体的平均得分与风险标记率；
- 发现显著偏差时，回退 Prompt 版本并重新评估。

### 6.3 持续校准

- 使用 LLM-as-a-Judge + 人工抽检评估 evidence 引用准确率；
- 目标：证据引用准确率 ≥ 85%，结构化输出合规率 ≥ 95%。

## 七、法律合规

### 7.1 员工知情权

在正式使用前，企业应向员工告知：

- 系统收集的数据类型与用途；
- 评估结果将如何被使用；
- 员工可查看的内容范围；
- 申诉渠道与流程；
- 数据保留期限与删除权利。

### 7.2 人工最终决策声明

所有 AI 输出页面必须展示：

> “本评估由 AI 生成，仅供参考，最终人事决策由主管/HR 人工作出。”

### 7.3 数据留存

| 数据类型 | 建议保留期限 | 删除条件 |
|---|---|---|
| 原始输入 | 2 年 | 员工离职满 1 年且无法律纠纷 |
| 评估结果 | 5 年 | 超过保留期限且完成合规审批 |
| 审计日志 | 5 年 | 超过保留期限 |

### 7.4 地区法规适配

- 中国大陆：遵循《个人信息保护法》《数据安全法》；
- 欧盟：遵循 GDPR，支持员工访问权、更正权、删除权；
- 其他地区：根据当地劳动法和数据保护法规调整知情同意书与数据留存策略。

## 八、红队测试建议

企业在上线前应执行以下安全测试：

| 测试类型 | 方法 | 通过标准 |
|---|---|---|
| Prompt 注入 | 在输入中嵌入越狱指令 | 输出不被操纵，不泄露管理视图 |
| 偏见测试 | 使用不同性别/年龄标签的相同输入 | 评分差异在可接受范围内 |
| PII 泄露 | 输入含敏感信息，检查输出 | 敏感信息被脱敏 |
| 权限绕过 | 尝试以 employee 角色访问管理视图 | 返回 403 |
| 审计完整性 | 删除或篡改审计日志 | 日志不可被非授权修改 |

---

## 九、安全运营建议

- 定期更新基础镜像、Python 依赖、前端依赖；
- API Key、数据库密码使用 Vault / K8s Secret / 阿里云 KMS 等密钥管理；
- 启用 WAF 与 DDoS 防护；
- 对管理后台启用 MFA；
- 制定事件响应预案，包括数据泄露、模型输出失控等场景。
